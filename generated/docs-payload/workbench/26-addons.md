An **Add-on** is a tool that plugs into the workbench: it is provided globally, it reserves a small declared area inside a project, and it follows a common SOP so that every Add-on is operated the same way. *Add-on* is the term the session glossary owns ([session/00-overview.md](/session/overview/) — "a custom-folder tool … not a reserved namespace"); this chapter keeps **custom folder** as the descriptive phrase for the same thing and specifies its model — where an Add-on may write, the SOP it follows, its standard entry points, and the relationship between its CLI and its skill. The taxonomy that distinguishes a custom folder (a tool) from a convention (a content format) is in [12-folders.md](/specification/folders/), which also carries the diagram of how registered folders, conventions, custom folders, and the wiki relate.

## The Memo System Is an Add-on Like the Others

The single, authoritative statement about the memo system's place is made here, so it is not repeated inconsistently elsewhere: **the memo system is an Add-on — a custom folder like the others.** There is **one** custom folder model for every tool; the memo system is not a conceptual special case, only a familiar instance of the same model. It is generated by `memo-init` and is the **recommended default** way of working, which is why a real project almost always carries it, but the workbench *core* does not depend on it ([12-folders.md](/specification/folders/)). Other chapters that touch this — the folder split, the SOP signpost — point **here** rather than restating it.

## Tool Global, Data Project-Bound

The defining tension a custom folder resolves is that the **tool is global** while its **data is project-bound**. The model holds both at once:

- **Declaration and policy** go into the project's `.workbench/` area — a tool **MAY** reserve a small descriptor there for its project-specific configuration ([22-config.md](/specification/config/)).
- **Work data** goes into the project's existing content folders — `data/` for raw input, `context/` for processed material ([12-folders.md](/specification/folders/)) — and, as the normal mode, flows **through the memo system** rather than into a private store.
- A custom folder **MUST NOT** claim its own new top-level project folder for its data; the registered-folder contract flags an unexpected top-level entry for review ([12-folders.md](/specification/folders/)).

The reservation is for *configuration and the specific*, not for bulk data. This also bounds what can be checked: the effective lint surface for a custom folder is its small `.workbench/` descriptor, **not** the gigabytes of data a tool may keep in its own global store. A tool needs an area to differing degrees — the memo system needs one for certain, a lighter tool may need only a descriptor or none.

## The Custom Folder Landscape

Custom folders span a wide size range, and the model is not forced uniformly onto all of them — the weight of the custom folder sets how much SOP it needs:

| Custom folder | Weight | Data | Custom steps |
|--------|--------|------|--------------|
| **memo system** | heavy | `.memo/` + `context/` | the full memo lifecycle |
| **a very large custom folder** | very large | a multi-GB global store; many provider keys | search/list → call |
| **a medium custom folder** | medium | a two-tier config | unlock → upload → format |
| **a small custom folder** | small/medium | a global store | a gate before an install |
| **a one-command custom folder** | small | none (stdout only) | one command |

A heavy custom folder earns a deep SOP with real Setup, Health, Update, and several Extras; a one-command tool needs little more than a single entry point. The point is one frame at all sizes, scaled to weight — not a separate model per tool.

## The Per-Custom-Folder SOP

Every custom folder follows the same four-part SOP — the common denominator the workbench skills already use ([24-skills-scope.md](/specification/skills-scope/)):

| Part | For a custom folder |
|------|---------------|
| **Setup** | Bring the custom folder into its expected state for a project (link the CLI, write the `.workbench/` descriptor). |
| **Health** | Report whether the custom folder is correctly installed and reachable, and whether its declared state is intact. |
| **Update** | Roll a central improvement to the custom folder out to the projects that consume it. |
| **Extras** | The tool-specific capabilities — the only part that differs between custom folders. |

The **frame is identical** across custom folders; only the Extras are tool-specific. The integration between a custom folder and the workbench falls into a few recurring **cascades** — *data* (where its material lands), *gate* (a check it enforces, such as a pre-install gate), *config* (what it declares in `.workbench/`), and *knowledge* (what it contributes to the wiki). Naming the cascade kind makes a custom folder's footprint predictable.

## Standard Entry Points — Health and Migrate

Two entry points are **normalized across all custom folders** so that tooling can drive any custom folder without special-casing it:

- **Health** — a uniform "is this custom folder in order?" check.
- **Migrate** — a uniform "bring this custom folder's project data to the current shape" step (for many custom folders this slot is currently empty; that is acceptable).

Existing tools expose these under divergent names (`memo health`, and the various tool-specific status/info/session verbs a custom folder already ships). Those names are kept as **aliases** of the standard entry points; the normalization is to a common vocabulary, and renaming an existing tool is deliberately **deferred** rather than forced now.

## CLI Is the Standard Entry, the Skill Orchestrates

A custom folder is reachable two ways, with a clear division of labor:

- The **CLI** (`<tool> health`, `<tool> migrate`) is the **standard entry point** — deterministic, machine-checkable, and **greppable**. Because the call is a plain command, an after-the-fact validator can detect from the transcript that it ran ([20-cli.md](/specification/cli/), [25-validation-overview.md](/specification/validation-overview/)).
- The **skill** is the **orchestrator** above it — the AI-facing entry point that sequences steps, asks the right questions, and calls the CLI to do the deterministic work.

**Standardization happens at the CLI level**, because that is the layer a validator can verify mechanically. The skill provides judgment; the CLI provides the checkable, repeatable action. This is the same split the workbench applies to its own skills ([24-skills-scope.md](/specification/skills-scope/)) and the seam the runtime call-validation measures against.
